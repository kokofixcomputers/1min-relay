## Триггер (команда пользователя)
Если в сообщении владельца есть команда **`/git`** (как отдельное слово или в начале строки) — **обязательно** следуй этому skill (workflow GitHub: gh, clone, ветки, commit, push, PR, инструмент правок по правилам ниже). Не игнорируй `/git`: сначала **напиши в чат**, что активирован skill **dev-github**, затем выполняй шаги.


## Выбор инструмента правок (обязательно по тексту запроса)
Смотри **текущее сообщение пользователя** (не только `/git`):

- Если есть слово **`cursor`** (без учёта регистра: Cursor, CURSOR) — **только Cursor Agent CLI** (`agent` на сервере). **Не** используй для правок кода `claude` в этом запросе.
- Если есть слово **`claude`** (Claude, CLAUDE) — **только Claude Code** (`claude` на сервере). **Не** используй для правок кода `agent` в этом запросе.
- Если **оба** слова есть — выбери инструмент по **последнему** упоминанию в сообщении; если неясно — спроси одним коротким вопросом, чем править.
- Если **ни одного** слова нет — по умолчанию **Cursor Agent CLI** (`agent --trust --model auto --print -p "..."`), если `agent` доступен; иначе `claude` с моделью из этого skill.

Команды (напоминание):
- Cursor: `cd <repo> && agent --trust --model auto --print -p "<задача + тест-план>"` (ask/plan: `--mode ask` / `--mode plan`).
- Claude: `cd <repo> && claude --model glm-4.7 --print "<задача + тест-план>"`.


## Сообщения в чат во время работы (обязательно)
Пользователь должен видеть прогресс в Telegram/чате, а не «тишину на минуты».

1) **Каждый новый шаг** — короткое сообщение: что делаешь сейчас (1 строка), например: «Проверяю gh auth», «Клонирую в /root/dev-workspace/…», «Запускаю agent…», «Делаю commit».
2) Перед **долгой** операцией (клон большого репо, `agent`/`claude`, тесты) — напиши, что запускаешь и что может занять время.
3) **Heartbeat каждые 5 минут**: если шаг **всё ещё** выполняется и прошло ≥5 минут с последнего твоего сообщения пользователю — отправь служебную строку: «⏳ Всё ещё в работе: кратко что именно (прошло ~N мин)». Продолжай до завершения шага или ошибки.

(Таймер реализуешь самосознанием: отслеживай длительность; для реально мгновенных команд heartbeat не нужен.)


## GitHub аутентификация и типичные ошибки (pct 121)
- PAT в `/root/CURSOR/github_pat.token`. После смены токена:  
  `tr -d '\r\n' < /root/CURSOR/github_pat.token | gh auth login --with-token -h github.com`
- Сразу после логина или при странностях с `git pull`/`git fetch`: **`gh auth setup-git`** — чтобы **git** использовал тот же токен, что и **gh** (иначе возможен **401** при приватных репо: в `git` остаётся старый пароль/токен в credential helper).
- Для приватных репозиториев предпочитай **`gh repo clone OWNER/REPO путь`** вместо «голого» `git clone` с устаревшими сохранёнными учётными данными.
- Переменная для скриптов: `GH_TOKEN` (из файла, не из чата).
- **401 / Invalid username or token** при `git` при рабочем `gh auth status` — почти всегда: не выполнен `gh auth setup-git`, либо нужно обновить PAT и снова `gh auth login --with-token`.


## Назначение
Безопасный workflow: клонирование, ветка, правки выбранным инструментом, проверки, commit, push, PR.


## Обязательные правила безопасности
- Не печатай токены/секреты/полные конфиги.
- Перед коммитом проверяй diff на секреты.
- Не коммить секреты.


## Рекомендуемый workflow (строго)
1) Рабочая папка: `/root/dev-workspace/<repo>`.
2) Клон: `gh repo clone OWNER/REPO` (или после исправления auth — `git clone` с рабочим helper).
3) Ветка: `e2e/<описание>-YYYYMMDD-HHMM`.
4) Правки — **только** выбранным по разделу «Выбор инструмента» CLI (`agent` или `claude`).
5) Проверки: `python3 -m py_compile …`, при наличии — быстрые тесты.
6) Перед коммитом: `git status`, `git diff --stat`.
7) Commit, `git push -u origin HEAD`, при необходимости `gh pr create`.


## Issue → Fix → PR
- `gh issue view <n> --json title,body,labels`
- ветка `fix/issue-<n>-...`
- fix выбранным инструментом, тесты, PR со ссылкой на issue.
